Evaluated absorbing the **obra/superpowers** Claude Code plugin into dot-llm (sdlc flavor). The full content of its 14 skills was read.

**Layer analysis.** superpowers = the *execution-discipline* layer (TDD, debugging, verification, code review — *how* to work). dot-llm = the *knowledge/structure* layer (intake → specs → plans → archive — *what* the project is). They are complementary on execution and overlapping/conflicting on planning, ideation, and session-bootstrap.

**Decision: do NOT enable the plugin.** Enabling it whole imports `using-superpowers` — a `SessionStart`-hook bootstrap that forces skill-discovery before any response — which competes with dot-llm's loading-rule kernel for control of session entry; and most of its skills overlap existing pillars/skills. It is multi-harness (NOT Claude-only) and MIT, so absorbing the *text* is clean.

**Consensus mechanism — the "discipline", a new third artifact type**, sitting between a slogan-directive (eager, always-on, dilutes) and a Claude Code skill (harness discovery). A discipline is a modular `disciplines/<name>.md` file in the flavor, **declared in `domain.md` and pulled into context by the loading rule by relevance** (eager index line in domain + lazy body) — the same eager-index/lazy-body profile as a skill, but native to `.llm/` and portable across harnesses. Each file: frontmatter `source.url` + `source.license` (MIT attribution mandatory) + `applies-when`; body = **Gate** (one invariant) + **Cycle** (steps, only if it's a process) + **Red flags**.

**Why discipline over skill or directive:** it uses dot-llm's own loading rule (one mechanism, not a parallel harness system); loads by relevance (no always-on dilution, unlike a directive); and allows the detailed cycle (not a slogan). Mechanics confirmed against official docs: a skill body is lazy / its description eager; `CLAUDE.md`/directive is fully eager; **neither is enforced — only a hook enforces**. The discipline is guidance carried by the loading rule, like everything else in `.llm/`.

**Absorb (as disciplines):** `verification` (gate), `systematic-debugging` (cycle), `test-driven-development`, `receiving-code-review`. `requesting-code-review` → an *artifact* (`code-reviewer.md` rubric for a subagent), not a discipline.
**Reject:** `using-superpowers` (kernel conflict), `writing-plans` / `executing-plans` (overlap `llm-plan`; flat-file format incompatible with `plans/<KEY>/`), `brainstorming` (overlap `llm-explore`), `finishing-a-development-branch` + `using-git-worktrees` (auto-destructive git vs read-only policy + `archive` overlap; env already has `EnterWorktree`), `subagent-driven-development` as a skill (duplicates Lead/Dev — lift only the progress-ledger + model-selection ideas). `writing-skills` → lift technique only ("Match the Form to the Failure"; "description = trigger, not workflow summary").

**Tooling:** a `skill-to-discipline` skill lives in the top-level `skills/` (repo-maintenance, NOT flavor-shipped) and carries the conversion recipe, so future plugins can be absorbed the same way.

**Status (2026-06):** five disciplines landed via `skill-to-discipline` — `disciplines/{verification,systematic-debugging,test-driven-development,receiving-code-review,acceptance-testing}.md` — plus the `## Execution disciplines` table in the sdlc `domain.md`; `skills/skill-to-discipline/SKILL.md` is the reusable converter. `doctor` green.

`acceptance-testing` was mined from the **xhyqaq/superpowers-plus** fork — its one genuinely new capability vs obra: an evidence-backed acceptance-criteria gate (a plan isn't done until every criterion has recorded a pass; a FAIL is a blocker, never weaken the criterion). It complements our authoring-only EARS; the plumbing (acceptance-tester subagent + playwright + `docs/superpowers/` paths) was rebound onto our intake/specs/archive.

Still pending from the plus: the `distilling-lessons` rubric (cold-read test + "When NOT to Apply" + mandatory commit-anchor) to harden `archive`→delta; and a subagent-dispatch discipline (auto parallel/serial routing + cheap-model default + minimal task-local context — supersedes the `dispatching-parallel-agents` idea). Reject the plus's `writing-acceptance-criteria` (flat `AC-NNN`, a regression vs our EARS) and its `docs/superpowers/` tree (overlaps our pillars). The plus is a defasado fork (v5.0.6 < obra 6.0.3) whose README oversells a non-existent `commits/` layer — cherry-pick concepts, don't adopt it. Pending from obra: `requesting-code-review` → a `code-reviewer.md` *artifact*.

**gstack garimpo (2026-06).** `kushal9889/gstack-cowork` (a repackaging of Garry Tan's `gstack`, ~57 huge skills) is mostly an operational CLI/browser engine — reject. Three genuinely-new methodology capabilities are PENDING (not pulled): `idea-validation` (from `office-hours` — a pre-intake "should this exist? is the problem real?" gate, the one stage upstream of intake), `plan-review` (from `plan-{ceo,eng,design,devex}-review` + `autoplan` — ONE discipline with 4 conditional lens-checklists; Eng = always-on ship-gate), and `security-review` (from `cso` — whole-repo audit orthogonal to code-review; its AI/skill-supply-chain phases threaten dot-llm itself; port its 22-item false-positive-exclusion rubric verbatim). Rubrics to harvest into existing disciplines: `investigate`'s 3-strike / >5-file circuit-breakers → `systematic-debugging`; `spec`'s secret-scan + read-before-ask; `learn`'s prune/contradiction loop → `distilling-lessons`.

**Release report = `llm-release` (implemented; NOT a discipline).** gstack's quantitative `retro` was rejected, but it sparked `llm-release`: a release report narrated from a GitLab compare between two heads (the user provides FROM/TO or the skill asks). Built as the **intake mould** — skill `llm-release` + `templates/release-report.md` + `skills/llm-release/scripts/gitlab-compare.sh` (curl+jq on the compare API) — output a standalone `.md` (no pillar/schema). Decision of record: a parametrized API-fetching report generator is a TOOL (skill+script), never a discipline (disciplines are runtime-free prose). Smoke-tested offline (jq extraction + usage/env guards + doctor 0/0/6) and via a real install (the `/llm:release <FROM> <TO>` slash command + the skill with its script land in `.claude/`); the live GitLab fetch itself is unverified (needs a real token/project). `/llm:release` resolves/confirms the two refs (lists tags/branches when ambiguous) before fetching; the script also extracts delivered tracker keys (regex `[A-Z][A-Z0-9]+-[0-9]+`) from commit titles.

**PENDING — "Requested vs delivered" reconciliation (Renato's idea).** Cross the delivered keys (from commits) against what was requested, into 3 buckets: delivered-as-requested / requested-not-delivered / **delivered-beyond-scope** (the "delta a mais" — keys delivered but not requested, plus un-keyed commits). "Delivered" is already extracted; the open decision is the **source of "requested"**: the local `intake/` mirror (offline, but has no version field) vs a **direct tracker query by version** (e.g. Jira `fixVersion = <TO>` — fiel, needs the version field + an intake-adapter extension). Sub-decisions: key regex (Jira-style default), and whether un-keyed commits count as beyond-scope.

**`skill-to-discipline` is the standing absorption mechanism for THIS repo** (Renato confirmed) — mirrored into `.claude/skills/` via a relative symlink to the canonical `skills/` source (single source, no drift). Caveat: `.claude/` is gitignored here, so the symlink is local to this checkout — a fresh clone must recreate it; only the `skills/` source is versioned. See [[dot-llm-frameworks-layout]], [[dot-llm-universal-index-and-loading-rule]].

**How to apply:** when a new plugin is interesting, run `skill-to-discipline` per skill; convert only execution-discipline skills, reject anything that governs session-start or overlaps a pillar. Keep the `domain.md` list as prose/table (NOT a path-list tag) so `llm doctor`'s file-refs check doesn't bind to it.
